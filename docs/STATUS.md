## In progress

- **In Codex Mode B review: `fix/ban-realtime-revoke`** тАФ **security fix (AUTH-1)** from the 2026-07-01 codebase audit. Community chat is delivered over Supabase Realtime Broadcast, authorized by `public.chat_topic_is_member` (`supabase/realtime-auth.sql`), which gated on **membership + live-community only** тАФ it never checked `users.banned_at`/`deleted_at`, and a ban can't revoke the target's Supabase session by user id (P-8). So a just-banned user's still-valid JWT kept authorizing `chat:{communityId}` subscriptions and **receiving live Article 9 messages** until the token expired. **Fix:** the predicate now `JOIN public.users` and requires `banned_at IS NULL AND deleted_at IS NULL`, denying a banned/erased identity at the next subscribe/setAuth тАФ mirroring the HTTP `isAuthenticated` gate at the Realtime layer (only tightens; active members unaffected; preserves the SECURITY DEFINER / empty-search_path / topic-injection hardening). **Applied to TEST + PROD** (verified the guard is present; live exposure closed). **Regression test** `server/__tests__/realtime-auth.integration.test.ts` calls the predicate directly as the `authenticated` role (JWT-claims GUC): activeтЖТallow; banned/erased/non-member/deleted-communityтЖТdeny (5/5). **Residual (P-8):** an already-open pre-ban socket flows until the next auth check / token expiry тАФ instant kill needs revoke-by-userId. **Codex Mode A: approved** with 2 accepted refinements (robust test-harness GUC/role setup; DEPLOY.md checklist adds banned/erased cases). No app-table / API contract change.
